# 中国 GUI 外观设计专利交底与 Word 成稿工作流

## 目标

默认按中国大陆 GUI 外观设计专利口径处理。最终交付物通常是：

1. 标准 `02界面说明.docx` Word 文档；
2. 与 Word 中状态图一致的单独 JPG 图片包。

但最终成稿不是第一步。这个 skill 必须先完成材料收集、图片理解、草稿确认、单图 review，再进入 Word 组装。

## 总原则

- 先引导，后起草；先确认，后生成文件。
- 用户只说“我想做一个 XXX 的 GUI 专利”时，也应启动本 skill，并进入冷启动引导。
- 不要把本地目录、IDE 打开的文件、历史草稿默认当成用户已授权材料；只能把它们作为“可用候选材料”询问是否使用。
- 在用户确认进入成稿阶段前，不要创建 Word、JPG 包、成稿包、预填包、补图清单文件。清单和草稿可先在聊天中给出。
- 每轮提问尽量一次收拢，最多 3 个关键问题；不要连续追问碎片信息。
- 总图用于理解流程，单张状态图用于最终 Word 和 JPG 图片包。

## 工作阶段总览

按下面顺序推进，除非用户明确要求只做某个局部任务：

```text
Phase 1  冷启动与材料收集    ← 用户说出产品/专利主题
Phase 2  图片/材料分析       ← 用户上传总图、多张截图、PRD 或交底稿
Phase 3  专利说明草稿        ← AI 输出摘要、状态说明、交互说明，用户 review
Phase 4  单张状态图收集核查  ← 用户上传单图，AI 逐一 review
Phase 5  Word 组装           ← 用户确认成稿后生成 Word + JPG
Phase 6  最终 Review 与交付  ← AI 自检并向用户输出复核摘要
```

## Phase 1：冷启动与材料收集

当用户只给出主题或产品名称时，先确认三类信息，不要直接分析或生成文档：

1. 专利对象：这是哪个产品、平台、功能或业务流程的 GUI；
2. 流程范围：希望保护哪些主要页面或用户路径；
3. 当前素材：是否有流程总图、单张截图、录屏帧、PRD、交底稿、Word 模板。

冷启动阶段不要追问这些信息：

- 申请人、设计人、身份证号、联系方式；
- 是否请求保护色彩；
- 是否需要相似设计或拆案；
- 复杂法律策略判断。

这些问题在看到图、确认状态集合或进入 Word 信息表阶段后再处理。

推荐提问口径：

“这是中国大陆 GUI 外观设计专利吗？你现在手里有这个流程的总图、单张产品截图，还是只有文字说明？先把现有材料发我，我会先做图片/流程分析，不会直接生成 Word。”

如果只有文字材料：可先帮用户整理应补哪些图，但不能进入最终成稿。

如果发现本地已有疑似材料：先问“我看到工作目录里有 XXX，是否以它作为本案材料？”用户确认前不得使用其内容推进。

## Phase 2：图片/材料分析

用户上传总图、拼接图、多张截图、录屏帧或说明文档后，先分析内容并输出结构化核对结果，不要直接写专利正文。

分析输出固定使用这个结构：

```markdown
## 图片识别结果

- 识别到的界面状态数量：X 张
- 排列依据：从左到右 / 从上到下 / 按标注顺序 / 按用户说明

| 临时编号 | 界面状态名 | 来源图 | 流程角色 | 初步判断 | 待确认点 |
| --- | --- | --- | --- | --- | --- |
| 图01 | XXX | 总图第 1 张 | 起始状态 | 建议保留 | 无 |
| 图02 | XXX | 总图第 2 张 | 等待态/异常态/分支状态 | 待确认 | 是否纳入主流程 |

## 流程路径理解

[用一小段说明主流程、可选分支、等待态、异常态。]

## 需要确认

- [一次性列出无法从图片直接判断的问题。]

## 下一步材料缺口

- [说明是否还需要单张状态图、哪些图需要重传或补充。]
```

分析时必须覆盖：

| 项目 | 要求 |
| --- | --- |
| 初步状态集合 | 按顺序列出识别到的界面状态 |
| 流程路径 | 说明主流程、可选分支、等待态、异常态 |
| 每图判断 | 标注建议保留、待确认、建议剔除 |
| 关键疑问 | 一次性列出需要用户确认的问题 |
| 下一步材料缺口 | 说明是否还需要单张状态图 |

输出后必须请用户确认状态集合和流程顺序。未经确认，不进入 Phase 3。

## Phase 3：专利说明草稿

在用户确认状态集合后，才把图和流程转化为模板正文草稿。正文只包含标准 Word 需要的内容：

- 摘要；
- 设计要点；
- 各界面状态标题；
- 各界面状态说明；
- 交互说明。

不要默认输出“案件摘要、风险提示、申请策略、视图计划、可申请性分析”等模板外报告，除非用户明确要求。

逐图说明只写截图可见事实：

1. 当前界面处于什么状态；
2. 主要区域；
3. 可见元素；
4. 当前界面的可执行操作。

避免主观评价、技术实现、算法逻辑、后台逻辑、营销语言。

草稿输出后必须进入草稿 review，询问用户确认：

- 状态名称和顺序是否准确；
- 每张图的界面描述是否符合产品实际；
- 交互路径是否遗漏、误解或需要删改。

用户确认或修改完成前，不进入 Phase 4。

## Phase 4：单张状态图收集与 Review

最终 Word 必须插入单张状态图，不能只使用总拼图、流程图或带箭头说明图。

单图上传时机：

1. 状态集合已确认；
2. 正文草稿已确认或基本确认；
3. Word 组装前。

如果用户只给总图：引导用户按已确认状态逐张上传单图，或明确从总图拆出可单独使用的状态图。

提示用户上传单图时，直接列出完整清单：

```text
进入文档组装前，需要以下 X 张单张状态图（PNG 或 JPG）：
1. 变化状态图01_[状态名称]
2. 变化状态图02_[状态名称]
3. 变化状态图03_[状态名称]
...

请逐张上传或打包上传。图片不要包含红色箭头、说明文字、蓝色选框、流程标题等交底标注。
```

用户上传单图后，必须逐一 review，并输出核查表：

| 正式编号 | 状态名称 | 图片文件 | 对应关系 | 图面质量 | 结论 |
| --- | --- | --- | --- | --- | --- |

Review 要点：

- 图片是否为单张状态图；
- 是否与已确认状态一一对应；
- 是否存在红色箭头、说明文字、蓝色选框、流程标题等交底标注；
- 是否裁切掉关键导航、标题、步骤条、按钮或提示；
- 图片比例、清晰度、色彩是否一致；
- 是否存在重复、缺失或顺序错误。

有存疑图片时，用“待处理”说明原因；不得带着存疑图片进入 Word 组装。

全部通过后，必须再次确认：“X 张单张状态图全部核查通过。现在是否进入 Word 文档组装？”

## Phase 5：Word 组装

只有在以下条件都满足时，才生成正式 Word 和 JPG 图片包：

- 用户已确认状态集合；
- 用户已确认正文草稿；
- 单张状态图已齐且 review 通过；
- 申请人、设计人、日期、联系方式等模板字段没有硬缺口，或用户明确允许留空/待补；
- 用户明确要求或确认“生成 Word/输出成稿”。

标准模板优先。默认使用 `assets/word-template.docx`；如用户提供同结构 `02界面说明.docx`，按用户模板原结构填报。

Word 硬规则：

1. page01 只放封面标题和蓝字提示；
2. page02 只放信息表；
3. page03 才开始 `界面说明如下：`；
4. 保留模板分页符、分节符、标题顺序和图片位置；
5. 正文只保留模板已有栏目；
6. 状态图编号按最终保留集合连续重排；
7. Word 中插入的必须是 review 通过的单张状态图；
8. 同步导出命名一致的单独 JPG 图片包。

详细模板规则见 [references/company-word-template-rules.md](references/company-word-template-rules.md)。

### Phase 5A：脚本化导出（推荐，避免版式漂移）

在进入最终交付前，优先使用仓库自带的 Python 导出器把「结构化 JSON + 单张截图」组装成与公司模板一致的 DOCX，并同步导出 JPG 图片包：

1. 在本仓库创建虚拟环境并安装依赖：

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

2. 准备 UTF-8 JSON（字段约定见 [references/intake-and-output-schema.md](references/intake-and-output-schema.md)），并把 `interface_states[].image_path` 写成 **相对 JSON 文件所在目录** 的可读路径。
3. 执行导出：

```bash
.venv/bin/python scripts/export_gui_patent_docx.py \
  --input ./case.json \
  --template ./assets/word-template.docx \
  --output-dir ./dist \
  --docx-name 02界面说明.docx
```

说明：

- `--template` 可替换为用户提供的公司实例 `02界面说明.docx`，只要结构与 [references/company-word-template-rules.md](references/company-word-template-rules.md) 一致（封面 + 信息表 + `界面说明如下：` 正文锚点）。
- 导出器会在 `--output-dir/jpg/` 生成 `变化状态图01_*.jpg` 命名规则的 JPEG，用于「图片需单独提供 JPG」交付。
- Windows 亦可使用 PowerShell 一键导出：`.\scripts\build-case-packet.ps1 -InputPath .\case.json -ExportDocx`。

只有在模板锚点字段被破坏、模板与公司实例不一致的特殊情况下，才退回手工在 Word 内就地替换。

## Phase 6：最终 Review 与交付

生成后先自检，再向用户输出最终 review 摘要，不要只做内部 checklist。

自检至少包括：

- page01 / page02 / page03 结构是否正确；
- `界面说明如下：` 是否从第三页开始；
- Word 内是否全部为单张状态图；
- 是否残留模板原案例文案；
- 是否残留被删除状态或空项目符号；
- 状态图编号是否连续；
- Word 图片数量是否与正文状态一致；
- JPG 图片包数量和命名是否与正文状态一致。

最终 review 输出应包含：

1. 已生成的文件路径；
2. 状态图数量、编号和图片一致性结论；
3. 仍需用户/代理机构复核的待补字段或法律判断。

## 阶段跳过规则

只有这些情况可以跳过前置阶段：

- 用户明确说“已有确认稿，直接按这个生成 Word”，且材料包含单张状态图；
- 用户上传的材料本身已经包含明确状态编号、单张图和确认后的正文；
- 用户明确要求只做某一局部任务，例如“只 review 单图”或“只改交互说明”。

除此之外，按 Phase 1 到 Phase 6 顺序推进。

## 删除或重排状态

用户要求删除某一步时：

- 删除该步骤标题、说明、图片和单独 JPG；
- 后续状态图从 `变化状态图01` 重新连续编号；
- 交互说明同步删除该步骤；
- 检查正文中不得残留被删状态名称。

等待态、loading 态、失败态、空状态、弹窗态默认标为“待确认/可选”，不要自动放入主流程，除非用户确认它是保护重点。

## 不允许的行为

- 未经用户确认，把本地目录、IDE 打开文件或历史草稿默认当成案件材料。
- 用户未确认进入成稿阶段前，创建 Word、JPG 包、成稿包、预填包或补图清单文件。
- 用总图、拼接图、带箭头说明图直接插入最终 Word。
- 在 Phase 2 之前输出逐图说明正文。
- 在状态集合和流程顺序未经确认时，锁定最终状态图集合。
- 在单张状态图未 review 通过时进入 Word 组装。
- 在正文中新增“案件摘要”“风险提示”“申请策略”“视图计划”等模板外一级标题。
- 代替用户或代理机构最终决定是否保护色彩、是否拆案、是否主张优先权；只能给出待确认建议。
- 写主观评价词、技术实现、后台逻辑、算法逻辑或营销语言。

## 网页 / 官网 GUI 特例分支

当用户表述命中以下任一触发条件时，先读 [references/web-gui-crop-workflow.md](references/web-gui-crop-workflow.md)，按那份文档的"长图切片 + 清理 + 接回主流程"路径走，不要直接把整张长图当作单张状态图：

- 关键词：`官网`、`官网首页`、`Landing Page`、`H5 长页`、`Web App 整页`、`网页 GUI`、`按模块切片`、`长图截图`、`长图裁剪`
- 上传单张纵向特长截图（高:宽 > 2:1 且高度 > 3000 px）
- 上传 Figma / Sketch / XD 的整页导出图作为 GUI 材料

主要工具：

- `scripts/crop_long_screenshot.py` — 通用长图分段切片，支持 `--auto-detect` 自动找候选分界 + 手动 `--segments` 配置精确切片
- `scripts/clean_cursor_artifact.py` — 清除截图中的鼠标光标、悬停残影、重复元素

切完得到单张状态图集合后，照常走 Phase 3-5。
