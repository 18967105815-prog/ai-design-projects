# 单张总图模式完整示例

本示例用于演示：当用户先上传 1 张包含全部 GUI 图的总图时，这个 skill 应如何识别结构，并最终过渡到标准 Word 成稿流程。

## 示例场景

用户说：

```text
我这里有一张总图，里面已经排好了这个 GUI 外观设计专利要用的全部图。你帮我整理成最终 Word 文档。
```

## 第一步：先做总图识别

先输出总图拆分识别结果，而不是直接写最终正文。

推荐输出：

```markdown
## 总图拆分识别结果

- 已识别子图数量：6
- 当前排序依据：按版面从左到右、从上到下
- 疑似最终保留状态图：子图01、子图02、子图03、子图04、子图05
- 疑似局部图或补充图：子图06
- 暂时无法确认的区域：无

### 子图清单

1. 子图01：支付首页
2. 子图02：确认付款
3. 子图03：输入密码
4. 子图04：支付处理中
5. 子图05：支付成功
6. 子图06：金额卡片局部放大图
```

## 第二步：收束最终保留状态图

在总图识别之后，要明确：

- 最终保留哪些状态图
- 哪些子图删除
- 是否需要局部图
- 状态图如何连续编号

这一步完成后，才能进入最终 Word 成稿准备。

## 第三步：补齐单张状态图

这是新工作流里的硬规则：

- 总图只用于识别结构
- 单张状态图才用于最终 Word 插图和 JPG 图片包

所以进入最终成稿前，必须让用户补齐每个最终保留状态对应的单张图。

推荐提示：

```text
当前总图已经足够用于识别最终保留状态，但在进入最终 Word 文档前，还需要补齐每个保留状态对应的单张图。最终 Word 和 JPG 图片包都必须使用单张状态图，不能直接插入拼接总图。
```

## 第四步：再起草最终正文素材

拿到单张图后，再写：

- 摘要
- 各界面状态说明
- 交互说明

不要在只有总图时就直接输出最终 Word 正文。

## 第五步：进入最终 Word

满足以下条件后，直接进入标准 `02界面说明.docx` 结构导出：

1. 最终保留状态图已确定
2. 每个保留状态已有单张图
3. 摘要、状态说明、交互说明已起草完成
4. 模板关键字段已具备或允许留空/待补充

## 这个示例想表达什么

单张总图模式的标准顺序现在应该始终是：

1. 总图拆分识别结果
2. 最终保留状态图收束
3. 补齐单张状态图
4. 起草模板正文素材
5. 导出最终 Word 和 JPG 图片包

不再是旧的“拆图 -> 逐图分析 -> 审核 -> 可选文档组装”逻辑。
